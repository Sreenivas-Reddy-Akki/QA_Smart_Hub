Manual Testing: Beginners to Advanced

Your complete guide to Quality Assurance principles and practices.

1. What is Software Testing?

Software testing is the process of checking whether an application works correctly and meets business requirements. It helps find defects before release so the product is stable, reliable, and user-friendly.

Simple Example (Login Page)

  1. Check valid user can log in.
  2. Enter invalid password — verify correct error shown.
  3. Verify the “Forgot Password” link navigates correctly.

Purpose of Testing

  • Ensure software behaves as expected.
  • Find defects early.
  • Improve product quality and user satisfaction.
  • Build customer confidence.

2. Why is Testing Important?

Testing helps to mitigate risk, maintain quality, and avoid business losses. It validates both functionality and non-functional expectations like performance and security.

Key Benefits

  • Identify bugs early — reduce fix cost.
  • Improve performance and security posture.
  • Ensure cross-device and cross-browser compatibility.
  • Protect business reputation and revenue.
Example: If an e-commerce app allows negative prices due to a bug, the company could lose money. Testers prevent such issues by validating inputs and flows.

3. Software Development Life Cycle (SDLC)

SDLC is the structured process used to design, develop, test, deploy, and maintain software.

Common SDLC Phases (step-by-step)

  1. Requirement Gathering: Document what users/business need.
  2. Design: Create architecture and UI/UX designs.
  3. Development: Engineers implement the code.
  4. Testing: QA verifies functionality and quality.
  5. Deployment: Release software to users.
  6. Maintenance: Patch bugs and release updates.
Example (Banking App): Requirement → Customer can view balance. Design → Balance screen layout. Development → API and UI for balance. Testing → Verify shown balance matches backend.

4. Software Testing Life Cycle (STLC)

STLC defines the sequence of testing activities to ensure quality is built into the product.

STLC Phases (step-by-step)

  1. Requirement Analysis: Identify testable requirements and clarify ambiguities.
  2. Test Planning: Define scope, resources, schedule, and deliverables.
  3. Test Case Design: Write test cases and test data for each requirement.
  4. Test Environment Setup: Prepare hardware, software, and data.
  5. Test Execution: Run test cases and record results.
  6. Defect Reporting: Log defects with steps, severity, and priority.
  7. Test Closure: Evaluate exit criteria, prepare summary reports, and archive artifacts.
Example (Login Page): Requirement → verify login. Test cases → valid login, invalid login, password reset. Execution → run across browsers. Defect → log and retest until closed.

5. Test Plan, Test Case, and Test Scenario

Test Plan

A document outlining what to test, how, and who will test. Typically includes objectives, scope, strategy, resources, schedule, and entry/exit criteria.

Test Case (step-by-step format)

Test Case ID: TC001
Title: Verify login with valid credentials
Preconditions: User account exists
Steps:

Open app

Enter valid email

Enter valid password

Click Login
Expected Result: User lands on dashboard
Actual Result: (to be filled during execution)
Status: Pass / Fail

Test Scenario

A high-level idea of what to test. Example: Verify user login functionality (covers multiple test cases like valid, invalid, and boundary conditions).

Practice Tip: Create at least 10 test cases for login covering valid, invalid, empty, boundary, and security inputs.

6. Defect / Bug Life Cycle

A defect is a mismatch between expected and actual results. The bug life cycle tracks progress from discovery to closure.

Common States (step-by-step)

  1. New: Tester logs the defect.
  2. Assigned: Lead assigns it to a developer.
  3. Open / In Progress: Developer starts work.
  4. Fixed: Developer marks as fixed and adds details.
  5. Retest: Tester verifies the fix.
  6. Closed: If fixed, defect is closed.
  7. Reopen: If issue persists, tester reopens it.
Example: Tester reports "Login button not clickable". Log as New → Assigned to Dev → Fixed → Tester retests → Closed.

7. Severity and Priority

These fields help decide the order and urgency for fixing defects.

Definitions

  • Severity: Technical impact of defect (e.g., crash, data corruption).
  • Priority: Business urgency (e.g., payment failing vs minor UI typos).

Examples

  • App crash on checkout → High Severity, High Priority.
  • Typo on marketing page → Low Severity, High Priority if on home page for campaign.

8. Testing Techniques

Common techniques used to design tests based on knowledge of the system.

Black Box Testing

Test without internal code knowledge — focus on inputs and outputs.

Example: Enter letters into phone number field and expect validation error.

White Box Testing

Developer-level testing that validates internal logic and code paths.

Example: Unit tests to ensure loops and branches execute correctly.

Grey Box Testing

Tester has partial knowledge of internals which helps create better test data.

Example: Knowing DB schema to validate input edge-cases efficiently.

9–11. Functional & Non-Functional Testing Types

Functional Testing

Validates specific functions/features work as required.

  • Login/Logout
  • Search
  • Payment gateway

Functional Sub-types (quick steps)

  • Smoke Testing: Quick checks to see if build is testable (sanity).
  • Sanity Testing: Verify specific bug fixes or small areas after change.
  • Regression Testing: Re-run tests to ensure changes did not break features.
  • Re-Testing: Verify defects once fixed.
  • Exploratory Testing: Ad-hoc investigations to find unexpected issues.
  • UAT (User Acceptance Testing): End-user validation before go-live.

Non-Functional Testing Types

  • Performance: Speed and stability under load.
  • Load Testing: Expected concurrent user checks.
  • Stress Testing: Push beyond limits to find breaking points.
  • Security Testing: Vulnerabilities like SQL injection, XSS.
  • Usability: Ease-of-use and navigation.
  • Compatibility: Cross-browser and cross-device checks.
  • Accessibility: Screen readers, keyboard navigation for differently-abled users.

12. Agile Testing

In Agile, testing is integrated into development sprints for continuous feedback.

Key Practices (step-by-step)

  1. Join requirement discussions early.
  2. Create story-based test cases during the sprint.
  3. Perform continuous testing (unit + UI + exploratory) as features are developed.
  4. Attend daily stand-ups to surface risks.
  5. Participate in sprint retros to improve test processes.
Example: In a 2-week sprint, implement login — testers write and execute cases within the sprint; devs fix defects quickly and tester retests before demo.

13. Shift-Left Testing

Shift-left means moving testing earlier in the lifecycle so defects are caught sooner and cheaper.

  1. Review requirements and design for gaps.
  2. Create test cases before code is written.
  3. Use unit tests and early integration tests.
  4. Use mocks/stubs to validate functionality before full integration.
Example: Tester finds a missing validation in requirements for mobile screens before development — saves rework.

14. Risk-Based Testing

Prioritize testing efforts on features with the highest business or technical risk.

How to do it (step-by-step)

  1. Identify features and their business value.
  2. Estimate likelihood of failure and impact.
  3. Rank features by combined risk score.
  4. Allocate more test effort to high-risk items.
Example (Banking App): Login and transfer functions = high risk → exhaustive tests. Help page = low risk → minimal tests.

15. Root Cause Analysis (RCA)

RCA finds the underlying reason a defect occurred so it can be prevented in the future.

RCA Steps (structured)

  1. Collect facts: logs, screenshots, test steps.
  2. Reproduce the issue reliably.
  3. Brainstorm possible causes with the team.
  4. Use methods like 5 Whys or Fishbone diagram to dig deeper.
  5. Implement corrective and preventive actions.
Example: Crash on Save → RCA finds missing null-check. Fix → add validation and update tests to cover null case.

16. Test Metrics and Reporting

Track metrics to measure testing progress and product quality.

Common Metrics & Formulas

Metric Formula Example
Test Case Execution % (Executed / Planned) × 100 (80 / 100) × 100 = 80%
Defect Leakage (Prod defects / Total defects) × 100 2 / 20 = 10%
Test Pass Rate (Passed / Executed) × 100 (70 / 80) = 87.5%
Report Example: Total Test Cases: 120; Passed: 110; Failed: 10; Defects Logged: 8; Defects Closed: 5.

17. Test Management Tools

Tools that help organize test artifacts, execution tracking, and defect management.

  • JIRA: Manage user stories, tasks, and defects.
  • TestRail / Zephyr: Store and execute test cases.
  • Bugzilla: Open-source defect tracker.
Example Workflow: Create story in JIRA → Add test cases in TestRail → Execute tests and log defects back in JIRA.

18. API Testing (Manual)

API testing validates backend endpoints without the UI — tools like Postman are commonly used.

Steps to Test an API

  1. Understand API spec (endpoint, method, params, headers).
  2. Create test requests with valid and invalid inputs.
  3. Validate status codes, response body, headers, and performance.
  4. Log issues for incorrect behavior or error codes.
Example (GET): 
GET https://reqres.in/api/users/2

Expected: Status = 200, response contains name = "Janet"
Example (POST): 
POST https://reqres.in/api/users


Body: { "name": "John", "job": "Tester" }
Expected: Status = 201 (Created)

19. Test Closure Activities

Finalize testing, archive artifacts, and gather lessons learned.

Step-by-step Closure Checklist

  1. Ensure all planned test cases are executed or documented as deferred.
  2. Confirm defects are closed or properly deferred with business sign-off.
  3. Prepare a Test Summary Report (metrics, risks, and recommendations).
  4. Archive test artifacts (cases, data, environment details).
  5. Conduct a lessons-learned session and update test assets.
Example Summary: Modules tested: 5; Total bugs: 30; Fixed: 28; Deferred: 2; UAT Signoff: Done.

22. Practice Exercises for Learners

  1. Write 10 test cases for a login page — include valid, invalid, empty, and boundary values.
  2. Perform exploratory testing on a demo site (e.g., demoqa.com) — note any unexpected behavior.
  3. Create a bug report for any issue you find — include summary, steps, expected, actual, severity, and priority.
  4. Design a test plan for an e-commerce checkout module — include scope, test types, and risks.
  5. Test REST API endpoints using Postman — validate status, payload, and error handling.

23. Summary

Manual testing is a foundational skill for QA professionals. It builds domain understanding, sharpens user-focused thinking, and complements automation by catching issues that automated tests may miss.

Final note: Practice writing clear test cases, reporting concise bugs, and collaborating with developers — these skills scale your impact as a QA professional.

24. Interview Questions & Resources

Use these resources to prepare for interviews and validate your core knowledge.

Key Interview Prep Resources

  • ISTQB Certification: Standardized concepts for core knowledge.
  • Online Testing Communities: Websites and forums for real-world scenarios.
  • Practice Websites: Use sites like The Sauce Demo Site for test case generation.

Top 5 Interview Questions

  1. Explain the difference between Severity and Priority. (Key: Severity is impact, Priority is urgency).
  2. Walk me through the Defect Life Cycle. (Key: New -> Assigned -> Fixed -> Retest -> Closed/Reopen).
  3. What is Regression Testing and why is it important? (Key: Re-verification after change to prevent breaks).
  4. How do you perform Boundary Value Analysis for an input field? (Key: Test min, max, min-1, max+1).
  5. Describe your approach to Exploratory Testing. (Key: Ad-hoc, concurrent learning and design, not script-based).
